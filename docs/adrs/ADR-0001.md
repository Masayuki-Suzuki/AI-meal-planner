# ADR-0001: パッケージマネージャーにpnpmを採用

## Status
Accepted

## Context
新規プロジェクトを単一リポジトリでスタートするにあたり、将来的なモノレポ化・
プロジェクト拡大を見据えてパッケージマネージャーを選定する。

## Decision
単一リポジトリのスタート時点では npm / yarn / pnpm の速度・ディスク差はほぼなく
実益は小さい。しかし将来のモノレポ移行とプロジェクト拡大後に依存関係を正しく
保つ観点から pnpm を採用する。

pnpm の strict な依存解決による phantom dependency 対策の利点が大きく、宣言して
いない依存を開発時点で検出できるため、後から発生しうる不要なデバッグにリソースを
割かずに済む。互換性リスクは許容範囲内で、まれに必要になる public-hoist-pattern
での対応も運用コストは小さいと判断した。

## Consequences
Pros:
- スタート段階から phantom dependency を防止できる
- モノレポ移行が他PMより比較的容易(workspaces + Turborepo/Nx が pnpm を前提に整備)
- strict な依存分離がデフォルトで効き、依存ツリーが実態を正直に反映する

Cons:
- フラットな node_modules を前提とする一部パッケージ(約1〜2%)で互換性の摩擦が出る

将来見直すトリガー:
- 依存パッケージの互換性問題が頻発し、public-hoist-pattern での対応が常態化した場合
- プロジェクトが単一リポジトリのまま長期固定され、pnpm の利点が実益を生まなくなった場合

## Alternatives considered
- npm: 互換性は最も高く安定。ただし phantom dependency を防げず、依存関係の
  「正しさ」を担保できない点で、拡大を見据えた今回の選定基準では pnpm に劣ると判断。
- yarn v4 (Berry): PnP による strict さと zero-install は強力だが、PnP はツーリング
  互換性の摩擦が最も大きく(一部パッケージは node_modules フォールバックが必要)、
  導入・運用の学習コストが高い。将来モノレポ化する場合でも pnpm workspaces の方が
  摩擦が少ないため今回は不採用。
